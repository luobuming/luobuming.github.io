32位，NX和Full RELRO

main函数首先从/dev/urandom文件中读取4个字符到buf中

/dev/urandom生成随机数的速度要快很多，一般就使用/dev/urandom获取随机数。

sub_804871F函数中让输入一个数，然后用strlen函数来获取我们读入的长度，然后再用strncmp函数来比较，首先我们知道strlen函数遇到\x00字符就停止了，倘若我们第一个输入的就是\x00那么岂不是就比较0个字节，然后就恒等，绕过去了

sub_80487D0函数用于读取，读取的数量得看参数得值，而传入这个函数的参数时上一个函数的返回值，上一个函数的返回值我们可以控制，进而就可以造成任意地址输入，造成可以覆盖ret地址，（最大长度是\xff），然后就是平常ROP拿到shell

book manager

这道题说实话并不难，但是程序的流程我一开始分析的透彻，导致我花了一天的时间，自闭

保护全开，64位程序

有八个选项：

在Add text中存在长度的溢出，输入长度只限制了不能大于0x100，没有其他限制，导致堆溢出，我们可以利用这个点来leak libc

最鸡肋的地方在输入函数sub_DAF，编辑前要用memset清0，长度就是要编辑的长度，所以我们要伪造后面的chunk

另一个漏洞点就是在update中的修改Text处，

这里有一个0xff长度的输入，但是只能edit第一个book的第一个section，这里很是难受，对于这里就要去前面之前构造chunk的时候要想到后门的需求

攻击流程：

  • 利用第一个溢出来leak libc
  • 然后利用第二个溢出构造触发Unsorted Bin Attack，在__free_hook的上方写上main_area+88地址
  • 利用double free，创建一个在__free_hook上方的chunk达到可以控制__free_hook指针，然后覆盖成system的地址
  • 调用free，来释放一个名字叫/bin/sh的chunk，达到getshell
